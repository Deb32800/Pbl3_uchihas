Smart Parking Navigation System

Requirements Document

Preface

NavPark was developed to solve real-life parking issues in the urban areas of Hong Kong (Yau Tsim Mong district). Initially, we aimed to recommend routes and parking lots based on both traffic and availability. However, due to difficulties in obtaining historical traffic data, we decided to focus primarily on parking availability prediction, while relying on external APIs for routing.

This requirements document reflects that shift in focus. It outlines how users will interact with the app, from viewing an interactive map and setting destinations, to receiving real-time updates and personalized parking suggestions. Our goal is to make the parking experience smoother and less time-consuming by providing relevant, up-to-date information in a user-friendly way.

Through this document, we aim to provide a clear structure for how the app should function, based on the features we believe will most benefit users in a realistic and implementable way.

Abstract

The NavPark app is designed to enhance urban parking experiences by recommending parking lots based on real-time availability predictions rather than proximity alone. Instead of merely guiding users to the nearest lot, the app intelligently suggests the closest available parking space within walking distance of the user's destination. While routing is handled through external APIs to ensure reliability and efficiency, the core innovation of NavPark lies in its focus on predicting parking lot vacancy to reduce the time and stress associated with finding parking in congested urban areas.

Objectives

The app attempts to fix and mitigate several issues found around cities with a large population:

  • Allow users to find a parking lot easily in an overcrowded city
  • Predicting parking lot availability to improve decision-making before and during a trip
  • Minimizing unnecessary traffic caused by cars circling for parking
Glossary
Term Definition
Interactive Map A touch-responsive map in the app that shows real-time traffic, parking locations, and other interactive elements.
Congested Roads Roads with heavy traffic, shown on the map to help users avoid delays.
Cache Temporary data storage that allows the app to function even when the network is unavailable.
Thresholds Numeric rules used to classify parking availability (e.g., green = available, red = full).
Map Pin Selection A method for users to set destinations by tapping on the map instead of typing an address.
Walking Distance The distance from a parking lot to a selected destination, used to recommend nearby lots.
Optimized Route A calculated driving route that considers traffic, distance, and parking availability to find the best path.
Historical Availability Past data showing how full parking lots were at different times, used to predict future availability.
Peak-Time Fees Additional parking costs applied during busy periods like weekday mornings or holidays.
Special Event Pricing Temporary changes in parking fees due to local events such as concerts or festivals.
Push Notification App alerts that notify users of key events like arrival, departure time, or changes in availability.
Localization The ability of the app to support multiple languages and adapt to regional formats.
Anonymized Data Data stripped of personally identifiable details before storage or analysis, to ensure privacy.
Prediction Model A system using machine learning to estimate future parking availability based on data.
One-Handed Use UI design principle ensuring major app functions are reachable using one thumb.
Temporary Files Short-term files like cached images stored on the device and automatically deleted after a set time.
Scheduled Navigation A feature allowing users to pre-plan a route for a future time, with reminder alerts.
Concurrent Users The number of people actively using the app at the same time.
Scalability The system's ability to support growing numbers of users without performance issues.
Route Logging Storing data about routes taken or suggested, used for analytics and improvement.
Availability States Visual indicators (green/yellow/red) showing how full each parking lot is.
Real-Time Events Live factors such as road closures or holidays that influence parking recommendations.
Departure Time Input A feature for scheduling the navigation based on a chosen future departure time.
Backend Server The part of the app that runs on remote computers to process data, handle requests, and manage the database — not directly seen by users.
User Interface (UI) The layout and design elements users interact with, such as buttons, icons, and menus.
System Uptime The percentage of time the app is available and functioning properly without interruptions.
Location Update Requests When the app asks the user's phone or GPS for current location.
User Requirements
Functional Requirements (FR)

FR 1. Users shall be able to see an interactive map when they open the application.

  • FR 1.1: Users shall be able to interact with parking lots on the map to view detailed information.
  • FR 1.2: The system shall display roads in different colors—grey, orange, and green—to indicate varying traffic conditions.

FR 2. Users shall be able to visually understand parking availability using a color-coded system (green, yellow, red).

  • FR 2.1: Users shall be able to view a legend explaining the color codes for both traffic and parking availability.

FR 3. Users shall be able to input their current location or travel destination using various methods:

  • FR 3.1: Enable GPS-based location detection.
  • FR 3.2: Allow users to place a marker (drop a pin) on the map.
  • FR 3.3: Allow users to enter a location or destination via address input.
  • FR 3.4: Allow users to input their location using voice commands.
  • FR 3.5: Users shall be able to save frequently visited destinations for quick access.
Nonfunctional Requirements (NFR)

NFR 1. Users shall experience fast and responsive performance at all times.

  • NFR 1.1: Users shall be able to load the application in no more than 3 seconds.
  • NFR 1.2: The user shall see a refreshed screen after 60 seconds to display the most recent information.
  • NFR 1.3: The system shall respond to user input in no more than 1 second.
  • NFR 1.4: Once a user asks for a route, system response time for route generation shall not exceed 10 seconds under normal working conditions.
  • NFR 1.5: The system shall be able to provide the user with a recommended route in no more than three taps from when the app is opened.

NFR 2. Users shall be able to download and install the application without excessive storage use.

  • NFR 2.1: The app shall be less than 100 megabytes in size.
System Requirements
Functional Requirements

FR 1. Users shall be able to see an interactive map when they open the application.

  • FR 1.1: The system shall render an interactive map interface upon launch, showing real-time data.
  • FR 1.2: The system shall color-code roads (grey, orange, green) based on real-time traffic conditions.
  • FR 1.3: The system shall allow map interaction to display information about each parking lot when clicked.

FR 2. Users shall be able to visually understand parking availability using a color-coded system (green, yellow, red).

  • FR 2.1: The system shall provide a visible legend explaining color meanings for traffic and parking availability.
Nonfunctional Requirements

NFR 1. Users shall be able to access the application from their smartphones.

  • NFR 1.1: The system interface shall maintain consistent usability across Android and iOS.
  • NFR 1.2: The app shall be operable in both portrait and landscape orientations.

NFR 2. The design of the application should be clear and user-friendly.

  • NFR 2.1: All major user actions shall require no more than 3 interactions from the home screen.
