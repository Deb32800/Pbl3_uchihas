Smart Parking Navigation System

Requirements Document

Preface
1.1. Background

NavPark was developed to address real-world parking challenges in Hong Kong’s Yau Tsim Mong district. Originally, the app aimed to recommend parking based on both availability and traffic. Due to difficulties obtaining traffic data, we shifted focus to predicting parking lot availability, while using external APIs for routing.

This requirements document outlines that revised scope, detailing how users interact with the app—from setting destinations and viewing parking options to receiving real-time suggestions. Our goal is to provide a user-friendly solution that simplifies parking through relevant, accurate, and actionable information.

Abstract

NavPark enhances urban parking by predicting real-time lot availability near a user’s destination. Instead of guiding users to the nearest lot, it recommends available spaces within walking distance. Routing is handled via external APIs to ensure efficiency. The app’s key innovation lies in reducing parking time and stress through smart availability prediction, making it a practical solution for congested urban environments.

Objectives

The app attempts to fix and mitigate several issues found around cities with a large population:

  • Allow users to find a parking lot easily in an overcrowded city
  • Predicting parking lot availability to improve decision-making before and during a trip
  • Minimizing unnecessary traffic caused by cars circling for parking

User Requirements
5.1. Functional Requirements (FR)

FR 1. Users shall be able to see an interactive map when they open the application.

  • FR 1.1: Users shall be able to interact with parking lots on the map to view detailed information.
  • FR 1.2: The system shall display roads in different colors—grey, orange, and green—to indicate varying traffic conditions.

FR 2. Users shall be able to visually understand parking availability using a color-coded system (green, yellow, red).

  • FR 2.1: Users shall be able to view a legend explaining the color codes for both traffic and parking availability.

FR 3. Users shall be able to input their current location or travel destination using various methods:

  • FR 3.1: Enable GPS-based location detection.
  • FR 3.2: Allow users to place a marker (drop a pin) on the map.
  • FR 3.3: Allow users to enter a location or destination via address input.
  • FR 3.4: Allow users to input their location using voice commands.
  • FR 3.5: Users shall be able to save frequently visited destinations for quick access.

FR 4. After a destination is input, the map shall zoom in on that area and display all parking lots within a 500-meter radius.

  • FR 4.1: The system shall recommend one optimized route to a parking lot with available space.
  • FR 4.2: The system shall also display the shortest path to the nearest parking lot.

FR 5. Users shall be able to view detailed information about each parking lot, including:

  • FR 5.1: Opening hours
  • FR 5.2: Parking costs
  • FR 5.3: Expected parking availability by hour
  • FR 5.4: Estimated travel time from the inputted location to the parking lot

FR 6. Users shall be able to customize app behavior and display:

  • FR 6.1: Set preferred price range for parking.
  • FR 6.2: Set preferred distance from destination.
  • FR 6.3: Configure notification preferences.
  • FR 6.4: Select interface language (English or Japanese).

FR 7. Users shall be able to create an account requiring a username, phone number, password, and car information.

  • FR 7.1: Users shall be required to agree to terms and conditions which explains that their data will be gathered.

FR 8. Users shall be able to schedule navigation to begin at a selected time.

  • FR 8.1: Users shall be able to input a future arrival time to receive parking recommendations.
  • FR 8.2: Users shall be able to set a delayed start time for navigation.

FR 9. Users shall be notified when important information needs reporting.

  • FR 9.1: Users shall be notified when their selected parking lot becomes full.
  • FR 9.2: Users shall be notified when their selected parking lot’s parking availability changes.
  • FR 9.3: Users shall be notified when a closer parking lot becomes available.
  • FR 9.4: Users shall be notified when a better route becomes available based on traffic prediction.
  • FR 9.5: Users shall be notified when the scheduled departure time has been reached.
  • FR 9.6: Users shall receive a summary notification when they arrive at the parking lot.

FR 10. Users shall be able to provide feedback about the app.

  • FR 10.1: Users shall be able to provide feedback for incorrect traffic information.
  • FR 10.2: Users shall be able to provide feedback for incorrect parking availability.
  • FR 10.3: Users shall be able to provide feedback for incorrect parking lot information.
  • FR 10.4: Users shall be able to provide feedback for feature suggestions.

FR 11. Users shall be able to configure privacy settings to their own likings.

  • FR 11.1: Users shall be able to request the deletion of their parking history.
  • FR 11.2: Users shall be able to request the deletion of their account.
  • FR 11.3: Users shall be able to change their passwords with ease.
  • FR 11.4: Users shall have the option to turn their GPS on only when the app is open.
5.2. Nonfunctional Requirements (NFR)

NFR 1. Users shall experience fast and responsive performance at all times.

  • NFR 1.1: Users shall be able to load the application in no more than 3 seconds.
  • NFR 1.2: The user shall see a refreshed screen after 60 seconds to display the most recent information.
  • NFR 1.3: The system shall respond to user input in no more than 1 second.
  • NFR 1.4: Once a user asks for a route, system response time for route generation shall not exceed 10 seconds under normal working conditions.
  • NFR 1.5: The system shall be able to provide the user with a recommended route in no more than three taps from when the app is opened.

NFR 2. Users shall be able to download and install the application without excessive storage use.

  • NFR 2.1: The app shall be less than 100 megabytes in size.

NFR 3. Users shall be able to navigate the app easily and intuitively.

  • NFR 3.1: New users shall be able to understand the functionalities of the app in no more than 10 minutes.
  • NFR 3.2: Users shall be able to understand the functionalities of the app without a tutorial.
  • NFR 3.3: Users shall be able to navigate the app easily while only using one hand.
  • NFR 3.4: Users shall be able to navigate the app in either English or Japanese.
  • NFR 3.5: The system should provide onboarded tooltips for new users if desired.

NFR 4. Users shall be able to rely on the system to work consistently.

  • NFR 4.1: The app shall be available for users at all times excluding maintenance times.
  • NFR 4.2: The app shall not provide false information traffic and parking availability information to users.
  • NFR 4.3: Issues reported by the user must be fixed in no more than 3 days.
  • NFR 4.4: The app shall not slow down regardless of high concurrent usage.

NFR 5. Users shall be confident that their user data is being protected at all times.

  • NFR 5.1: Users should be automatically logged out after 120 seconds of inactivity.
  • NFR 5.2: Users should be notified when the application uses user data.
  • NFR 5.3: Users should be notified when there is an unusual activity in their account or application.

NFR 6. Users shall not have data related issues when a server or app side issue comes up.

  • NFR 6.1: Users should not lose their input when the application crashes.
  • NFR 6.2: The system shall ensure that user data is never lost even in the case of system failure.
  • NFR 6.3: The system shall automatically retry failed network requests up to three times before notifying the user.
  • NFR 6.4: The system shall display meaningful error messages to the user when failures occur, rather than crashing or freezing.
System Requirements
6.1. Functional Requirements

FR 1. Users shall be able to see an interactive map when they open the application.

  • FR 1.1: The system shall render an interactive map interface upon launch, showing real-time data.
  • FR 1.2: The system shall color-code roads (grey, orange, green) based on real-time traffic conditions.
  • FR 1.3: The system shall allow map interaction to display information about each parking lot when clicked.

FR 2. Users shall be able to visually understand parking availability using a color-coded system (green, yellow, red).

  • FR 2.1: The system shall provide a visible legend explaining color meanings for traffic and parking availability.

FR 3. Users shall be able to input their current location or travel destination using various methods:

  • FR 3.1: The system shall request access to the device's GPS to detect the user’s current location.
  • FR 3.2: The system shall allow users to drop a pin on the map to specify a location.
  • FR 3.3: The system shall allow users to type an address to input a destination.
  • FR 3.4: The system shall allow users to input a geographical location using voice inputs.
  • FR 3.5: The system shall allow users to save and manage frequently visited destinations.

FR 4. After a destination is input, the map shall zoom in on that area and display all parking lots within a 500-meter radius.

  • FR 4.1: The system shall recommend one optimized parking route based on distance, availability, and traffic.
  • FR 4.2: The system shall calculate and display the shortest route to the closest parking lot.

FR 5. Users shall be able to view detailed information about each parking lot, including:

  • FR 5.1: The system shall display the opening hours of each parking lot.
  • FR 5.2: The system shall display the parking cost of each parking lot.
  • FR 5.3: The system shall display expected hourly availability for each parking lot.
  • FR 5.4: The system shall calculate and display estimated travel time from the inputted location to the parking lot.

FR 6. Users shall be able to customize app behavior and display:

  • FR 6.1: The system shall allow users to set and store a preferred parking price range.
  • FR 6.2: The system shall allow users to define a maximum walking distance from destination.
  • FR 6.3: The system shall allow users to configure notification types and timings.
  • FR 6.4: The system shall support switching between English and Japanese for all UI elements.
  • FR 6.5: The system shall allow users to have tooltips if they find the application difficult to use.

FR 7. Users shall be able to see the current parking duration after parking their car.

  • FR 7.1: The system shall record the starting time once the user parked their car.

FR 8. Users shall be able to set expected parking duration and view estimated costs.

  • FR 8.1: The system shall calculate costs using hourly rates of the selected parking lot as well as any peak-time fees and special event pricing.
  • FR 8.2: The system shall display real-time updates if duration is adjusted.

FR 9. Users shall receive a summary notification upon arrival at the parking lot.

  • FR 9.1: The notification shall include parking lot name and walking distance to destination.
  • FR 9.2: The summary notification shall be sent once the GPS detects arrival.

FR 10. Users shall be able to save frequently visited destinations.

  • FR 10.1: The system shall allow manual tagging (e.g. "Work," "Home").
  • FR 10.2: The system shall support quick search for users to choose their frequently visited destination easily.

FR 11. Users shall input a future arrival time for parking recommendations.

  • FR 11.1: The system shall use historical data for the predictions.
  • FR 11.2: The system shall adjust recommendations based on real-time events such as public holidays.

FR 12. Users shall be able to set a delayed start time for navigation.

  • FR 12.1: The system shall allow input of departure time.
  • FR 12.2: The system shall send a push notification 10 minutes before the departure.

FR 13. Users shall be able to receive a notification at the selected departure time.

  • FR 13.1: The system shall support scheduling navigation for later departure.

FR 14. The system shall support scheduling navigation to begin at a later time as specified by the user.

  • FR 14.1: The system shall send a push notification to the user when their scheduled departure time is approaching.

FR 15. The system shall support easy account creation and information updating.

  • FR 15.1: The system shall require a simple account creation page including the input of information such as email address and password.
  • FR 15.2: The system should support information updating which can be carried out in no more than 2 minutes.
  • FR 15.3: Users shall be required to agree to terms and conditions which explains that their data will be gathered.

FR 16. The system shall be able to change their privacy settings according to their own likings.

  • FR 16.1: The system shall support account deletion and parking history deletion.
  • FR 16.2: Users shall be able to configure their geographical (e.g.: GPS) privacy settings.

FR 17. The system should provide a feedback section to allow users to voice their opinions of the application.

  • FR 17.1: Users shall be able to easily navigate to the feedback section of the application.
  • FR 17.2: Users shall be able to write any complaints about the application.

NFR 1. Users shall be able to access the application from their smartphones.

  • NFR 1.1: The system interface shall maintain consistent usability across Android and iOS.
  • NFR 1.2: The app shall be operable in both portrait and landscape orientations.

NFR 2. The design of the application should be clear and user-friendly.

  • NFR 2.1: All major user actions shall require no more than 3 interactions from the home screen.

  • NFR 5.1: All user data shall be anonymized before storage or analytics processing.

NFR 7. The system response time for route generation shall not exceed 10 seconds under normal conditions.

  • NFR 8.1: All primary navigation controls shall be placed at the bottom of the screen.

  • NFR 9.1: Extra features should be optional downloads.
  • NFR 9.2: The app should automatically clean up temporary files after a week.

NFR 10. The system shall maintain performance and reliability as user numbers grow.

  • NFR 10.1: The system shall support up to 10,000 concurrent users without performance plummeting.
  • NFR 10.2: The system shall be able to process 10,000+ location update requests per second without failure or lag.

System Evolution
Fundamental Assumptions:

This system assumes that reliable and up-to-date parking data can be accessed from external sources or APIs. It also relies on users granting location access and interacting with the app to input destinations. The current model operates under the assumption that users prioritize finding available parking spaces over simply reaching the nearest lot. It is also assumed that basic routing functionalities can be effectively outsourced to existing mapping services, allowing the project team to focus on vacancy prediction and UI/UX design.

Anticipated Changes:

In the future, the system may evolve to incorporate historical traffic data once it becomes available, enabling better routing functions. As more data is collected, the accuracy of parking availability predictions can also improve through machine learning. If possible, the system may also adapt to other cities beyond Hong Kong, requiring adjustments in data sourcing and UI localization.
